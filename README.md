# AWS 3-Tier Web Service 아키텍처 구축

서비스 안정성과 네트워크 격리를 목표로 한 AWS 기반 3-Tier 웹 서비스 인프라 아키텍처 설계 및 구축

---

## 🏗 Architecture

[ Route53 ]
                         │
                    [   ALB   ]
                         │
    ┌────────────────────┴────────────────────┐
    │              Public Subnet                │
    │         (Nginx Reverse Proxy)              │
    └────────────────────┬────────────────────┘
                         │
    ┌────────────────────┴────────────────────┐
    │             Private Subnet (WAS)          │
    │              (Apache Tomcat)               │
    └────────────────────┬────────────────────┘
                         │
    ┌────────────────────┴────────────────────┐
    │            Private Subnet (DB)             │
    │              (Amazon RDS - MySQL)           │
    └─────────────────────────────────────────┘

    Multi-AZ 구조로 가용 영역을 교차 배치하여 단일 장애점(SPOF)을 제거했습니다.

---

## 🛠 Tech Stack

`AWS Console` `VPC` `EC2` `ALB` `Nginx` `Apache Tomcat` `Amazon RDS (MySQL)` `Route53`

---

## 📌 주요 구현 내용

### 1. 네트워크 격리 설계
- VPC 서브넷 3계층(Web / WAS / DB) 구조 설계
- 계층별 보안그룹을 분리하여 트래픽 통제 및 라우팅 보안 강화
- DB 계층은 Private 서브넷 최안쪽에 배치하여 외부 접근 완전 차단

### 2. 고가용성 구성
- Multi-AZ 기반 EC2 교차 배치로 단일 장애점 제거
- ALB를 통한 트래픽 분산 처리

### 3. 리버스 프록시 연동
- Nginx - Tomcat 간 Reverse Proxy 통신 구성
- 정적 리소스와 동적 요청을 분리 처리하여 응답 효율 개선

## 👤 담당 역할

| 구분 | 내용 |
|---|---|
| 팀 프로젝트 | 인프라 설계 및 워크플로우 구성 담당. VPC 3계층 서브넷 구조 정의와 인바운드 보안그룹 정책 수립으로 팀 전체 구축 방향성 제시 |
| 독립 재구현 | 프로젝트 완료 후, 동일한 구조를 설계부터 배포까지 혼자 처음부터 단독 수행하며 전 과정 직접 확인 |

---

## 💡 트러블슈팅

재구현 과정에서 팀 프로젝트 때는 팀원이 이미 설정해둔 VPC CIDR 범위와 보안그룹 규칙을 그대로 사용했지만, 혼자 처음부터 구성하면서 각 설정값이 왜 그렇게 정해졌는지 직접 검증했습니다. 동작 여부 확인에 그치지 않고 설정을 바꾸거나 제거해보며 원인을 추적하는 방식으로 인프라를 이해했습니다.
